## Part 2: 当前解析器提取了什么

### 已提取字段

| 字段 | 来源实体 | 提取方式 |
|------|----------|----------|
| 产品名称 | PRODUCT | 正则匹配 `'name'` |
| Schema 类型 | FILE_SCHEMA | 正则匹配 |
| 是否装配体 | NEXT_ASSEMBLY_USAGE_OCCURRENCE | 存在性检查 |
| 整体包围盒 | CARTESIAN_POINT | 采样 5000 点计算 min/max |
| 零件数量 | MANIFOLD_SOLID_BREP | 计数 |
| 每零件面数 | CLOSED_SHELL → ADVANCED_FACE 引用 | 计数 |
| 曲面类型 | ADVANCED_FACE → surface 实体 | 正则匹配（采样50面） |
| 零件分类名 | 面数 + 曲面类型 | 规则推断 |
| 每零件包围盒 | 面数比例 × 整体尺寸 | **估算** |
| 零件颜色 | STYLED_ITEM → COLOUR_RGB 链 | 8步链追溯 |

### 未提取字段

| 字段 | STEP 中存在 | 未提取原因 |
|------|-------------|------------|
| **精确几何顶点** | 41,456 个 CARTESIAN_POINT | 需要按实体归属分组，正则无法可靠完成 |
| **面的精确边界** | 5,479 个 EDGE_CURVE | 需要拓扑遍历，正则无法完成 |
| **B样条曲面控制点** | 877 个 B_SPLINE_SURFACE | 需要解析参数化曲面数据 |
| **圆弧半径/圆心** | 331 个 CIRCLE | 可提取但当前未做 |
| **装配变换矩阵** | 无（非装配文件） | 文件本身不含 |
| **材料信息** | 无 | STEP AP214 不含 |
| **公差/粗糙度** | 无 | 文件本身不含 |
| **精确零件尺寸** | 需从几何计算 | 需要 3D 几何内核 |

### 提取率估算

```
几何信息（顶点/面/曲线）:   ~2%  （仅提取了包围盒，未提取任何顶点坐标）
拓扑信息（面/边/壳关系）:   ~15% （提取了面数和曲面类型，未提取面间关系）
装配信息:                   ~50% （仅判断是否装配体，此文件无装配关系）
材料信息:                   0%   （STEP 中不含）
颜色信息:                   ~80% （8步链追溯成功，但仅提取主色）
尺寸信息:                   ~10% （整体包围盒精确，零件尺寸为估算）
```

**综合提取率: ~15%**

## Part 4: 当前图片生成流程（无 Prompt）

**重要发现**: 当前图片生成**不使用豆包/任何 LLM**。完整流程为纯本地 Pillow 绘制。

```
STEP 文件
    ↓
step_parser.py（正则解析）
    ↓
ParsedProduct（名称/面数/曲面类型/颜色/估算尺寸）
    ↓
step_analysis_service.py（构建 ProductGraph）
    ↓
process_generation_service.py（生成步骤）
    ↓
instruction_service.py（组装 per_step_info）
    ↓
image_service.py（Pillow 程序化绘制）
    ↓
exports/images/step_N_xxxx.png
```

**image_service.py 实际绘制逻辑**:

```python
# 1. 固定虚拟尺寸（不使用实际尺寸）
v_length, v_width, v_height = 80, 60, 40
scale = 5.0

# 2. 根据曲面类型选择形状模板
if "圆柱面" and face_count <= 10:
    draw_cylinder(...)     # 画圆柱体
elif "球面" and "环面":
    draw_sphere(...)       # 画球体
elif "环面" and "圆锥面":
    draw_threaded(...)     # 画螺纹件
elif face_count <= 6:
    draw_plate(...)        # 画薄板
else:
    draw_3d_box(...)       # 画长方体

# 3. 用 STEP 颜色填充
fill_color = step_color_to_hex(part.color)

# 4. 不同步骤用不同视角
view = STEP_VIEWS[(sequence - 1) % 10]
```

**无 Prompt 发送给豆包。图片完全由 Pillow 代码生成。**

## Part 5: 分析 — 仅根据当前数据能否画出接近真实模型的图？

**结论: 不能。**

原因：

1. **无几何数据**: 当前传递给图片生成器的只有 `name`、`faceCount`、`surfaceTypes`、`color`、`length/width/height`。没有一个顶点坐标、没有一条曲线参数、没有一个面的边界。

2. **形状是模板而非真实**: "带孔支架"被画成一个长方体，"复杂主体"也被画成一个长方体。真实模型的复杂曲面（877个B样条曲面）完全无法用模板表达。

3. **尺寸是估算的**: 零件尺寸按面数比例分配，与真实尺寸可能相差数倍。

4. **空间位置未知**: 不知道零件在装配体中的相对位置，无法画出正确的装配关系。

5. **曲面细节丢失**: "圆柱面"只是一个标签，不知道圆柱的半径、高度、位置。

**类比**: 当前相当于只拿到了一份"零件清单"（名称、数量、大致尺寸），然后用简笔画把它们画出来。要画出真实模型，需要拿到完整的"施工图纸"（所有顶点坐标、曲线参数、面的拓扑关系）。

## Part 6: 导致图片不像真实模型的 Top 10 原因

按影响度排序：

| 排名 | 原因 | 影响度 | 说明 |
|------|------|--------|------|
| 1 | **无几何顶点数据** | ★★★★★ | 没有任何顶点坐标，无法构建真实形状 |
| 2 | **形状使用模板而非真实几何** | ★★★★★ | 圆柱/球体/长方体都是预设模板，与真实形状无关 |
| 3 | **无 B 样条曲面数据** | ★★★★☆ | 877 个自由曲面完全无法表达 |
| 4 | **不使用 AI 图片生成** | ★★★★☆ | 当前纯 Pillow 绘制，不调用任何图片生成模型 |
| 5 | **零件尺寸为估算值** | ★★★☆☆ | 按面数比例分配，可能偏差数倍 |
| 6 | **无零件空间位置信息** | ★★★☆☆ | 不知道零件在装配体中的位置和朝向 |
| 7 | **曲面类型仅作分类用** | ★★☆☆☆ | "圆柱面"标签无法表达具体参数（半径/高度/位置） |
| 8 | **颜色仅取主色** | ★★☆☆☆ | 一个零件可能有多种颜色（不同面不同色），当前只取一种 |
| 9 | **视角固定** | ★☆☆☆☆ | 10 个预设视角循环使用，非交互式 |
| 10 | **图片为 2D 投影** | ★☆☆☆☆ | 非真 3D 渲染，无光影/遮挡效果 |

## Part 7: 改进路线

### 方案 A: 最小改动（当前 MVP 可用）

**思路**: 不改变 Pillow 绘制方式，优化数据传递。

| 改进项 | 开发量 | 效果提升 |
|--------|--------|----------|
| 提取 CIRCLE 实体的半径/圆心 | 1天 | 圆柱体比例更准确 |
| 提取每个零件的精确包围盒（从 CARTESIAN_POINT 按归属分组） | 2天 | 尺寸标注准确 |
| 优化形状模板（增加更多形状变体） | 1天 | 视觉区分度提高 |
| 使用 Three.js 在前端做交互式 3D 预览（用包围盒数据） | 3天 | 用户可旋转查看 |

**总开发量**: ~7 天  
**风险**: 低  
**收益**: 图片仍为示意图，但尺寸更准确、形状更多样

### 方案 B: 中等改动（推荐）

**思路**: 引入轻量 3D 几何库，用真实顶点数据渲染。

| 改进项 | 开发量 | 效果提升 |
|--------|--------|----------|
| 用 `trimesh` 库解析 STEP（需 `opencascade` 后端） | 3天 | 获取真实几何网格 |
| 用 `pyrender` 或 `trimesh.scene` 渲染 3D 图片 | 2天 | 真实模型渲染 |
| 每步骤渲染不同装配状态 | 2天 | 展示装配进度 |
| 保留 STEP 颜色映射 | 1天 | 颜色准确 |

**依赖**: `trimesh[opencascade]`（约 200MB，pip 可装）  
**总开发量**: ~8 天  
**风险**: 中（opencascade 安装可能有平台问题）  
**收益**: 图片为真实 3D 渲染，与 STEP 模型一致

### 方案 C: 工业级方案

**思路**: 使用 pythonocc-core（OpenCASCADE Python 绑定）。

| 改进项 | 开发量 | 效果提升 |
|--------|--------|----------|
| conda 环境 + pythonocc-core 安装 | 2天 | 工业级 STEP 解析 |
| 提取完整 B-Rep 几何 | 5天 | 精确到每个面/边 |
| OpenCASCADE 3D 渲染 | 3天 | 专业级渲染质量 |
| 支持装配变换矩阵 | 3天 | 精确装配位置 |
| 支持 STEP 材质/纹理 | 2天 | 完整视觉效果 |

**依赖**: conda + pythonocc-core（约 500MB）  
**总开发量**: ~15 天  
**风险**: 高（conda 环境管理复杂，部署困难）  
**收益**: 工业级精度，可扩展为完整 CAD 查看器

## 总结

| 维度 | 当前状态 | 瓶颈 |
|------|----------|------|
| 数据提取 | ~15% | 正则解析器无法提取几何顶点 |
| 图片生成 | 纯 Pillow 模板绘制 | 无真实几何数据输入 |
| 颜色 | ~80% 准确 | 链追溯成功 |
| 尺寸 | ~10% 准确 | 估算值，非精确计算 |
| 与模型相似度 | ~5% | 仅形状分类（圆柱/球/板）有参考价值 |

**根本原因**: 当前架构是 `元数据 → 模板绘制`，而非 `几何数据 → 3D 渲染`。要生成接近真实模型的图片，必须获取并使用 STEP 文件中的几何顶点数据。
