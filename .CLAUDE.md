# 目标
定义 Claude Code 在本项目中的行为规范、工作方式和约束条件。
确保 Claude Code 以一致、高质量、可验证的方式完成所有开发任务。

---

# 核心原则

1. 合约优先（Contract First）
   - 所有实现必须遵循 05_CONTRACT.md
   - Contract 是唯一权威来源

2. 测试驱动（TDD）
   - 遵循 RED → GREEN → REFACTOR 流程

3. Epic 驱动（Epic Driven）
   - 每个 Epic 先做 Analysis，再做 Implementation Plan
   - 不跳过 Analysis 直接写代码
   - 不越过当前 Story 范围

4. 小步提交（Small Commit）
   - 每个 Story 完成后必须 Commit
   - 提交粒度：Story 级别
   - 格式：`<type>(<scope>): <description>`（Conventional Commits）

5. 可验证（Verifiable）
   - 所有代码必须有对应测试
   - 所有改动必须可验证

6. 中文化（Chinese First）
   - 所有文档、注释、docstring、UI 文本必须使用中文
   - 技术术语保留英文（如 ProductGraph、STEP、UUID、API、LLM 等）
   - 代码标识符（函数名、变量名）使用英文

# Claude Code 工作流程（必须遵循）

## Phase 1：Epic 分析

1. 阅读 Epic 描述
   - 理解 Epic 目标和范围
   - 识别依赖的 Feature 和 Story
   - 确认与其他 Epic 的边界

2. Architecture 分析
   - 对照 03_ARCHITECTURE.md 确认模块归属
   - 对照 04_DOMAIN_MODEL.md 确认聚合根
   - 对照 05_CONTRACT.md 确认 API 契约

3. 输出 Implementation Plan
   - 按 Feature 拆解 Story
   - 按 Story 拆解 Task
   - 每个 Task 明确 Inputs / Outputs / Acceptance Criteria

## Phase 2：按 Story 实现

4. 编写测试（RED）
   - 先写 Contract Test → Unit Test → Integration Test
   - 根据 Acceptance Criteria 编写失败测试
   - 运行测试确认失败

5. 实现最小代码（GREEN）
   - 编写最少的代码使测试通过
   - 不做额外优化
   - 只实现当前 Story 范围的功能

6. 重构（REFACTOR）
   - 优化代码结构
   - 消除重复
   - 提升可读性和可维护性

7. 验证
   - 运行全部相关测试
   - 确保全部通过
   - 确保 Coverage 达标

8. 文档更新（Documentation Update）
   - 更新相关文档（Architecture / Contract / Domain / Database / EPICS）
   - 确保文档与代码一致（遵循 ADR-007）

9. 提交（COMMIT）
   - 每个 Story 完成后 Commit
   - 编写清晰的 Commit Message
   - 格式：`<type>(<scope>): <description>`（Conventional Commits）

# 行为规范

必须：
- 每个 Epic 必须先做 Analysis 再进入实现
- 遵循 Testing Strategy
- 遵循 Repository Rules
- 遵循 Definition of Done
- 保持代码简洁、清晰、可读
- 每个 Story 完成后更新相关文档
- Contract 变更必须同步更新 05_CONTRACT.md

禁止：
- 不得跳过 Analysis 阶段
- 不得跳过测试
- 不得跳过 RED 阶段
- 不得跳过 REFACTOR 阶段
- 不得跳过 Documentation Update
- 不得实现超出当前 Story 范围的功能
- 不得修改无关文件
- 不得破坏 Contract
- 不得降低测试覆盖率

---

# 代码风格

- 遵循 PEP 8
- 类型注解必须完整
- 注释和 docstring 使用中文
- 函数和变量命名使用英文
- 单一职责原则
- 避免重复代码
- 优先使用纯函数

---

# 错误处理

- 所有异常必须明确处理
- 不得忽略异常
- 返回标准错误格式（见 Contract）
- 日志必须清晰

---

# 测试要求

- 每个 Story 必须有 Contract Test + Unit Test + Integration Test
- 单元测试覆盖核心逻辑
- 集成测试覆盖组件协作
- Contract 测试必须 100% 通过
- 关键领域必须 100% 覆盖

---

# 沟通规范

- 回答必须简洁、准确、结构化
- 代码必须可直接运行
- 解释必须基于项目文档
- 如有不确定，必须先提问
- 完成任务后，总结变更内容

# Git 提交规范

## 格式（Conventional Commits）
```
<type>(<scope>): <中文简述>

<body>（可选，说明 why 而非 what）
```

## Type 类型
| type | 含义 | 示例 |
|------|------|------|
| feat | 新功能 | feat(image): 渐进式装配参考图 |
| fix | 修复 bug | fix(test): BOM 测试后清理数据文件 |
| test | 测试相关 | test: 新增 20 个测试用例 |
| refactor | 重构 | refactor(bom): 移除硬编码数据 |
| docs | 文档 | docs: 更新 ROUTE_D 架构文档 |
| style | 格式/样式 | style: 前端布局调整 |
| chore | 杂项 | chore: 添加 .gitignore 规则 |

## Scope 范围（常用）
- `image` — 图片生成管线（reference_renderer, image_service, prompt_builder）
- `bom` — BOM 零件库（bom_library, bom_matcher）
- `frontend` — 前端（index.html, 静态资源）
- `test` — 测试文件
- `api` — API 端点（main.py, schemas）

## 规范
- subject 行不超过 72 字符
- 使用中文描述改动内容
- body 说明 **为什么** 改，而非改了什么
- 每次 commit 必须通过全部测试
- 不同功能分开 commit，不混在一起
- 不提交临时文件、缓存、IDE 配置

## 错误示例（避免）
```
❌ fix: 修复了一个问题          （太笼统）
❌ feat: update code            （英文 subject）
❌ fix: regex + color + bbox    （混多个改动）
```

## 正确示例
```
✅ fix(test): BOM 测试后清理数据文件，避免污染启动状态
✅ feat(image): 渐进式装配高亮参考图 - 已装配灰色 + 当前零件橙色高亮
✅ fix: regex double-quote support + color tuple handling
```
