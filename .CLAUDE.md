# 目标
定义 Claude Code 在本项目中的行为规范、工作方式和约束条件。  
确保 Claude Code 以一致、高质量、可验证的方式完成所有开发任务。

---

# 核心原则

1. 合约优先（Contract First）  
   - 所有实现必须遵循 05_CONTRACT.md  
   - Contract 是唯一权威来源

2. 测试驱动（TDD）  
   - 遵循 RED → GREEN → REFACTOR 流程

3. Epic 驱动（Epic Driven）  
   - 每个 Epic 先做 Analysis，再做 Implementation Plan
   - 不跳过 Analysis 直接写代码
   - 不越过当前 Story 范围

4. 小步提交（Small Commit）  
   - 每个 Story 完成后必须 Commit
   - 提交粒度：Story 级别

5. 可验证（Verifiable）  
   - 所有代码必须有对应测试  
   - 所有改动必须可验证

---

# Claude Code 工作流程（必须遵循）

## Phase 1：Epic 分析

1. 阅读 Epic 描述
   - 理解 Epic 目标和范围
   - 识别依赖的 Feature 和 Story
   - 确认与其他 Epic 的边界

2. Architecture 分析
   - 对照 03_ARCHITECTURE.md 确认模块归属
   - 对照 04_DOMAIN_MODEL.md 确认聚合根
   - 对照 05_CONTRACT.md 确认 API 契约

3. 输出 Implementation Plan
   - 按 Feature 拆解 Story
   - 按 Story 拆解 Task
   - 每个 Task 明确 Inputs / Outputs / Acceptance Criteria

## Phase 2：按 Story 实现

4. 编写测试（RED）
   - 先写 Contract Test → Integration Test → Unit Test
   - 根据 Acceptance Criteria 编写失败测试
   - 运行测试确认失败

5. 实现最小代码（GREEN）
   - 编写最少的代码使测试通过
   - 不做额外优化
   - 只实现当前 Story 范围的功能

6. 重构（REFACTOR）
   - 优化代码结构
   - 消除重复
   - 提升可读性和可维护性

7. 验证
   - 运行全部相关测试
   - 确保全部通过
   - 确保 Coverage 达标

8. 文档更新（Documentation Update）
   - 更新相关文档（Architecture / Contract / Domain / Database / EPICS）
   - 确保文档与代码一致（遵循 ADR-007）

9. 提交（COMMIT）
   - 每个 Story 完成后 Commit
   - 编写清晰的 Commit Message
   - 格式：`<type>(<scope>): <description>`（Conventional Commits）

# 行为规范

必须：
- 每个 Epic 必须先做 Analysis 再进入实现
- 遵循 Testing Strategy
- 遵循 Repository Rules
- 遵循 Definition of Done
- 保持代码简洁、清晰、可读
- 每个 Story 完成后更新相关文档
- Contract 变更必须同步更新 05_CONTRACT.md

禁止：
- 不得跳过 Analysis 阶段
- 不得跳过测试
- 不得跳过 RED 阶段
- 不得跳过 REFACTOR 阶段
- 不得跳过 Documentation Update
- 不得实现超出当前 Story 范围的功能
- 不得修改无关文件
- 不得破坏 Contract
- 不得降低测试覆盖率

---

# 代码风格

- 遵循 PEP 8
- 类型注解必须完整
- 函数和变量命名必须清晰
- 单一职责原则
- 避免重复代码
- 优先使用纯函数

---

# 错误处理

- 所有异常必须明确处理
- 不得忽略异常
- 返回标准错误格式（见 Contract）
- 日志必须清晰

---

# 测试要求

- 每个 Story 必须有 Contract Test + Unit Test + Integration Test
- 单元测试覆盖核心逻辑
- 集成测试覆盖组件协作
- Contract 测试必须 100% 通过
- 关键领域必须 100% 覆盖

---

# 沟通规范

- 回答必须简洁、准确、结构化
- 代码必须可直接运行
- 解释必须基于项目文档
- 如有不确定，必须先提问
- 完成任务后，总结变更内容
